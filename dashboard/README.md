# SMS送信分析システム - ダッシュボード仕様書 ＆ 開発ドキュメント

本ドキュメントは、「ENEOSモビリニア中国支店 SMS送信分析システム」のダッシュボード（配信スケジュール管理機能）における実装仕様、UX設計原則、および今後のシステム設計・要件定義に引き継ぐべきフィードバック事項をまとめたものです。

---

## 1. システム概要とアーキテクチャ

本ダッシュボードは、店舗ごとのSMS送信スケジュールおよび配信実績を視覚的かつ直感的に管理・編集するためのフロントエンドツールです。

*   **フロントエンド**: HTML5, CSS3 (Vanilla CSS), JavaScript (Vanilla JS)
*   **バックエンド / データベース**: Supabase (PostgreSQL)
*   **データ連携**: 配信実績CSVのアップロードによって、親キャンペーンレコードに紐づく配信実績（子明細）が登録・集計される仕組み。

---

## 2. データベース物理設計（キャンペーン関連）

### 2.1 テーブル構造と関連性

#### ① `campaigns`（配信親テーブル）
配信スケジュールの親となるキャンペーン情報を保持します。
*   `id`: `UUID PRIMARY KEY DEFAULT gen_random_uuid()`（主キー）
*   `delivery_date`: `DATE`（配信予定日）
*   `campaign_name`: `VARCHAR`（配信名。例：`20260422_オイル交換案内`）
*   `work_group`: `VARCHAR`（予約される作業。例：`事前点検`）
*   `work_type_id`: `VARCHAR`（**要件定義追加**: 作業種別ID）
*   その他、カテゴリ、基準、タイミング、送信文章内容等。

#### ② `sms_deliveries`（SMS配信実績テーブル）
CSVアップロードによってインポートされる、配信実績の明細データです。
*   `id`: `UUID PRIMARY KEY DEFAULT gen_random_uuid()`
*   `campaign_id`: `UUID REFERENCES campaigns(id) ON DELETE CASCADE`
*   `store_code`: `VARCHAR REFERENCES stores(store_code) ON DELETE CASCADE`
*   `hashed_customer_id`: `VARCHAR`（不可逆ハッシュ化した顧客ID）
*   `sms_count`: `INTEGER`（送信通数）

> [!IMPORTANT]
> **カスケード削除の仕様**
> `sms_deliveries` テーブルの `campaign_id` 外来キーには **`ON DELETE CASCADE`（連動削除）** 制約が設定されています。
> これにより、ダッシュボード上で「配信月アコーディオン」を削除、あるいは特定の親スケジュールを削除した際、関連する送信実績CSVデータもろともデータベース側で自動的かつ確実に物理削除され、ゴミデータ（孤立レコード）が残らない堅牢な設計となっています。

## 3. フロントエンド実装＆UX仕様 (特筆すべき設計)

本システムの開発において、業務システムの操作性とプレミアム感を極限まで高めるため、以下のUX仕様を実装しました。

### 3.1 アコーディオングループの「最上部ピン留め」ソート制御
*   **課題**: 配信月を新規追加したり年月ラベルを変更した際、標準の「降順ソート」が即座に働くと、編集中のアコーディオンが画面の下方へ流れてユーザーが見失うUXの不都合がありました。
*   **解決策**:
    *   画面上で追加・編集中の配信月キーをグローバル変数 `justAddedAccordion` で保持。
    *   ソート処理において、**「保存ボタンを押すまでは、新規作成・編集中の年月グループを最優先で一番上（最上部）に固定表示する」**制御を組み込みました。
    *   保存が成功した段階で初めてピン留めが解除され、正しい時系列（降順）へと滑らかに自動整列します。

### 3.2 保存ボタンの「インライン・ステートフィードバック」
*   **課題**: 保存完了時に画面右下にトースト（一時ポップアップ）が頻繁に出現する設計は、操作エリアを遮り、業務ツールとしての美観を損ねていました。
*   **解決策**:
    *   右下のトースト通知を完全に排除し、**「スケジュール変更を保存」ボタン単体で状態を表現**するインライン方式に統一しました。
    *   **保存中**: ボタンを非活性化 (`disabled`) し、`⏳ 保存中...` にテキストを変化。
    *   **保存成功**: ボタン背景色をブランドグリーンに変化させ、`✅ 保存完了！` を1.5秒間表示したのち元に戻します。
    *   **保存失敗**: ボタン背景色を警告レッドに変化させ、`❌ 保存失敗` を2秒間表示します。

### 3.3 完全な「空アコーディオン枠」の新設と一括作成テンプレート
*   **仕様**:
    *   「➕ 配信月を追加」をクリックした初期状態、および年月を切り替えた時点では、データベース側にダミーレコード等のゴミデータを一切登録せず、**スケジュールが完全に空（0件）の状態でアコーディオン枠のみを描画**します。
    *   空状態のグループ内には「💡 この配信月グループにはまだスケジュールが登録されていません。上のボタンから一括作成してください」という案内バナーと一括作成ボタンのみを表示します。
    *   「定期配信10本を一括作成」ボタンを押した瞬間に、定義された10本のスケジュールテンプレートがアコーディオン内に瞬時に展開されます。

### 3.4 タイムゾーン安全な日付解析
*   **仕様**:
    *   JavaScriptの `new Date("YYYY-MM-DD")` はブラウザのタイムゾーン設定（UTC/JST）によって日付が1日ズレたり、`Invalid Date` を引き起こす危険性があります。
    *   本システムでは日付文字列を `split('-')` で直接数値として分解し、年月日の加算を行う堅牢なロジックを実装することで、タイムゾーン依存のバグや日付計算ループによるフリーズを完全に防止しています。

### 3.5 配色の整理（カラーノイズの大幅カット）
*   **仕様**:
    *   画面内の強烈な「赤ボタン」や「緑ボタン」の並びによる視覚的ノイズを極限まで削減しました。
    *   グループ削除ボタンは普段は枠線のみの「アウトライン危険ボタン (`btn-outline-danger`)」とし、スプレッドシート行内の削除ボタンも普段は静かなグレーでホバー時にのみ薄赤になる「フラット削除ボタン (`btn-row-delete`)」を採用。業務ツールにふさわしい、モダンで落ち着いたプレミアムな配色（UX-Pro-Max基準）へ整理しました。

## 4. 要件定義および今後の開発へのフィードバック

今後のシステム開発（機能追加、別ダッシュボードの開発、API連携等）において、要件定義および基本設計にフィードバック・横展開すべき重要な教訓です。

### 4.1 JavaScript厳密比較における「データ型の不整合」への注意
*   **現象**: データベース側が `integer` 型（数値）のID、フロント側が `string` 型（文字列）のIDを扱っている場合、JavaScriptの厳密等価演算子 `c.id === id` は型不一致によりすべて `false` と判定され、編集や保存のイベントハンドラーが完全に無視されてしまいます。
*   **対策 (フィードバック)**:
    *   イベントハンドラー内やデータ同期時のID比較ロジックでは、常に **`String(c.id) === String(id)`** のように両者を明示的に文字列にキャストして比較する共通設計ルールを定義することを推奨します。

### 4.2 本番DB (Supabase/UUID) とクライアント側一時IDの競合対策
*   **現象**: クライアント側で新規作成した行を識別するため、`camp_17112345` のような文字列型の一時IDを付与してリストを管理します。しかし、この一時IDを含んだオブジェクトをそのまま Supabase の UUID型主キーを持つテーブルへ Upsert (登録) しようとすると、UUIDフォーマットエラーでデータベース側から保存が拒否されます。
*   **対策 (フィードバック)**:
    *   新規保存処理（API連携）を行う際、**「`id` が特定の一時プレフィックス（例：`camp_`）で始まる場合は、オブジェクトから `id` プロパティを除外して送信する」**というルールを共通実装する必要があります。これにより、Supabase側で自動的に安全なUUID主キー（`gen_random_uuid()`）が採番され、新規インサートが正常に完了します。

### 4.3 タイムゾーンを考慮した日付計算の標準化
*   **現象**: カレンダーから取得した日付文字列をそのままブラウザ標準の `Date` オブジェクトで処理すると、環境によっては前日の日付として登録される「日付ズレ」が発生します。
*   **対策 (フィードバック)**:
    *   業務ロジック内で配信日や起算日を扱う際は、ブラウザ依存のタイムゾーン変換をバイパスし、文字列をベースにした日付ユーティリティ（例：`YYYY-MM-DD` を `split('-')` して処理する簡易関数）をプロジェクト標準の共通モジュールとして要件・基本設計段階で定義しておくことが望ましいです。

### 4.4 親テーブル削除時のカスケードポリシーの必須化
*   **対策 (フィードバック)**:
    *   本システムのように「配信親（スケジュール）」と「配信実績（CSV明細）」が1対多の関係を持つ場合、親を削除した際に子が残ってしまうとデータ整合性が崩壊します。
    *   要件定義・DB物理設計の段階で、リレーションシップには必ず **`ON DELETE CASCADE`** を明示的に付与することをデータベース標準規約に盛り込んでください。これにより、フロントエンドは親のAPIを1回呼ぶだけで、関連する全ての実績レコードをクリーンに削除でき、API設計も極めてシンプルになります。
